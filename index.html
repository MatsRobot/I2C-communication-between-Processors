🔗 I2C-Bridge: Multi-Processor Data Exchange

A high-reliability communication framework bridging the Raspberry Pi Pico (Master) and Arduino (Slave). It features a robust dual-bus architecture to isolate sensor telemetry from display traffic, optimized for real-time robotic feedback loops.

Arbitration Fix: While I2C theoretically supports "Multi-Master," this project enforces a strict Master-Slave relationship via central clock management on the Pico, eliminating SCL synchronization errors.

Dual-Controller Hardware Architecture

How It Works: Dual-Bus Isolation Logic

This implementation utilizes the dual I2C controllers on the RP2040 to prevent data collisions and bus lag:

  • I2C0 (GP16/GP17) - The External Bus: Connects the Pico to the Arduino Slave (Address 0x0A) and a primary 0.96" OLED.
  • I2C1 - The Monitoring Bus: Dedicated to a secondary OLED display for local system status, ensuring visual updates don't interrupt sensor polling.
  • Cross-Platform Bridge: Handshakes between MicroPython (Pico) and C++ (Arduino) using readable text strings transmitted via onRequest interrupts.

Interrupt Safety

A critical architectural discovery: The Arduino cannot manage a local OLED while in Slave mode on the same bus without causing interrupt conflicts. All display tasks are delegated to the Pico Master.

Asynchronous Telemetry

The Arduino polls distance data locally and serves it instantly when the Pico triggers an I2C request, ensuring the Master never waits for a slow sensor reading.
